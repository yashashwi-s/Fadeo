# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Fadeo: a native macOS app that plays/switches/fades audio automatically based on what
you're doing (frontmost app, desktop/Space, meeting state, Focus mode, time). Full
product spec, architecture rationale, and decision log live in `PLAN.md` — read it before
making architectural changes, it is the source of truth, not this file.

## Build environment (read this first)

`xcode-select` on this machine points at Command Line Tools, which has **no XCTest and
cannot build the app target**. The full toolchain is Xcode, currently installed at
`/Applications/Xcode-beta.app`. The `Makefile` auto-detects a full Xcode via
`DEVELOPER_DIR` (prefers `/Applications/Xcode.app`, falls back to `Xcode-beta.app`) — use
`make`, not raw `xcodebuild`/`swift`, unless you export `DEVELOPER_DIR` yourself first.

```sh
make gen          # regenerate Fadeo.xcodeproj from project.yml (needed after adding/removing files)
make build         # build the app
make run           # build + launch from DerivedData
make install       # build + install to ~/Applications (Spotlight-visible) + launch — use this while developing
make test          # swift test on the FadeoCore package
make clean         # remove build/ and the generated .xcodeproj
```

Single test: `cd Packages/FadeoCore && DEVELOPER_DIR=/Applications/Xcode-beta.app/Contents/Developer xcrun swift test --filter ResolverTests/testAppInTwoWorkspaces_specificityWins`

`Fadeo.xcodeproj` is **generated by XcodeGen from `project.yml`** and is gitignored —
never hand-edit the `.xcodeproj`; edit `project.yml` and run `make gen`. Adding/removing
a Swift file requires regenerating the project before the next build sees it.

## Architecture

Two build products with a hard boundary between them:

- **`Packages/FadeoCore`** (SPM, zero dependencies, Swift 6 language mode) — the pure
  correctness core. No AppKit/Foundation-file-IO/OS calls. Everything here is a plain
  value type and is exercised by `swift test`, never by running the app.
- **`Fadeo/`** (Xcode app target, Swift 5 language mode) — all OS-touching glue:
  sensors, actuators, the config file, the UI. Consumes `FadeoCore` as a local package.

This split exists so the part that must be *provably correct* (precedence resolution)
never needs a Mac to test, and the OS glue stays thin and swappable.

### The pipeline (event-driven, no polling)

```
Sensor (OS push) → ContextPatch → AppController merges into Context
                 → debounced (Settings.evaluationDebounceMs)
                 → Resolver.resolve(Context, Config, ResolverState) → Decision
                 → Reconciler.reconcile(AudioState, target, transition) → AudioCommand
                 → InternalEngine.execute(command)
```

`AppController` (`Fadeo/Sources/App/AppController.swift`) is the only place that owns
this whole loop and mutable state (`Context`, `ResolverState`, `AudioState`). Sensors and
actuators are otherwise stateless with respect to each other.

### FadeoCore: the resolver is the core feature

`Resolver.resolve` (`Packages/FadeoCore/Sources/FadeoCore/Resolver.swift`) implements
**four ordered bands** — this is the precedence model and the most subtle part of the
codebase:

1. **Override** — any `Workspace.isOverride` workspace whose `match` holds pre-empts
   everything (e.g. "Meetings" pauses audio unconditionally).
2. **Candidates** — every enabled, non-override workspace whose `match` currently holds.
3. **Tiebreak chain** — when there are multiple candidates, `Settings.tiebreak` (default
   `[stickiness, specificity, priority, recency, stableId]`) narrows to one winner.
   `stickiness` means: if the currently-active workspace is still a candidate, keep it —
   this is what stops the audio from changing just because you tabbed through a shared app.
4. **Fallback** — no candidate matched; governed by `Settings.fallback`
   (`keepCurrent` / `resumePrevious` / `silence`).

`AppMembership.strength` (`strong` vs `weak`) matters here: a `weak` app can only
*preserve* the currently active workspace, never *activate* its own workspace. This is
the mechanism for "apps used across contexts (Slack, Notes) shouldn't yank you out of
whatever's currently playing" — see `Resolver.test` and the weak-membership tests in
`ResolverTests.swift` for the exact semantics before changing this logic.

Every `Decision` carries a `ResolutionTrace` (which band, which strategy, human
explanation) — this isn't cosmetic, it's what the "why is this workspace playing" UI and
future Conflict Simulator are built on, so don't drop it when refactoring.

`Reconciler.reconcile` is a second, separate pure function: it diffs `AudioState`
(what's actually playing) against the `Decision`'s target and emits the *minimal*
`AudioCommand` (`.none`/`.start`/`.crossfade`/`.setVolume`/`.stop`). This is what
prevents redundant fades/commands firing on every context tick — never bypass it by
calling the actuator directly from a Decision.

### Volume model: the system volume is the only master

There is deliberately no independent "Fadeo volume." `SystemVolume`
(`Fadeo/Sources/Platform/SystemVolume.swift`) reads/writes/observes the actual macOS
output volume via CoreAudio (`kAudioHardwareServiceDeviceProperty_VirtualMainVolume`,
event-driven via `AudioObjectAddPropertyListenerBlock`, no polling). `AppController`
mirrors it into `masterVolume`, and the menu-bar slider *sets the system volume* rather
than an app-local gain. `Workspace.sound.volume` / `perApp` overrides are a relative
*baseline mix* (per-source, per-app), separately perceptually calibrated in
`InternalEngine.calibratedGain` so equal baseline numbers sound equally loud across noise
textures (white noise is louder than brown at the same RMS). See PLAN.md §6a before
touching any of this — the "don't double-apply the system volume in software, and don't
add a second async master" constraint is easy to accidentally violate.

### Dual activation policy

`AppDelegate` (`Fadeo/Sources/App/AppDelegate.swift`) toggles
`NSApp.activationPolicy` between `.accessory` (menu-bar-only agent, no Dock icon, App
Nap eligible — the default/steady state) and `.regular` (Dock icon, full window) based on
whether a real app window is open. This is what lets Fadeo be both a full app with deep
preferences and an invisible background daemon; don't add a second window-management
path that bypasses `updatePolicy()`.

### Config: hot-reloaded, never crashes on a bad edit

`ConfigStore` (`Fadeo/Sources/Config/ConfigStore.swift`) is the single source of truth
for `Config`, persisted at `~/Library/Application Support/Fadeo/config.json`. It watches
the *directory* (not the file) via FSEvents (`FileWatcher`) so atomic editor saves
(write-temp-then-rename) are caught. On a decode failure it keeps the last-good config
and surfaces `lastError` instead of crashing — preserve this behavior in any change here.
It also guards against re-publishing on its own writes or byte-identical rewrites
(`lastWrittenData` check) — removing that guard reintroduces a re-evaluation storm (it
happened once during development; see git history around the M1 commit).

### Sensors

`Sensor` (`Fadeo/Sources/Sensors/Sensor.swift`) is a small protocol: own one OS signal,
emit `ContextPatch`es. Only `AppFocusSensor` (NSWorkspace activation notifications)
exists so far. New sensors (Space, meeting/camera-mic, Focus/DND) should follow the same
shape — push-based, zero cost when not started — and are intentionally *lazily
activated* in the eventual design (only start a sensor whose `ContextField`s some enabled
workspace actually references); `AppController.startSensors()` currently starts
everything unconditionally since there's only one sensor, revisit this when adding more.

### Actuators

`InternalEngine` (`Fadeo/Sources/Actuators/InternalEngine.swift`) is the only actuator so
far: an `AVAudioEngine` with a real-time render block that *synthesizes* ambient noise
(brown/pink/white via `NoiseRenderer`) rather than shipping audio files — no assets, no
looping seams, tiny footprint. Fades are sample-accurate gain ramps computed on the
control thread and applied per-sample on the audio thread (benign lock-free races by
design, worst case a one-sample discontinuity). The engine is fully `stop()`ed (not just
gain-zeroed) when idle so it drops to ~0% CPU and releases the audio hardware — always
route "silence" through `Reconciler`'s `.stop` command, don't just ramp volume to zero
and leave the engine running. External-player conduction (MediaRemote/AppleScript for
Spotify/Apple Music) doesn't exist yet (planned M2) — `AppController.applyAudio` currently
intercepts any `external:`-prefixed source and silences instead of pretending to play it.

## Efficiency is a hard constraint, not a nice-to-have

The whole point of the event-driven pipeline above is that Fadeo must be invisible in
Activity Monitor at idle: no polling loops, no repeating timers in steady state (the one
exception planned is a schedule sensor's single next-boundary timer), tear down audio
resources rather than muting them. When adding anything that runs periodically, default
to "this should be push-based" and treat a polling loop as something to justify, not the
default. See PLAN.md §11 ("the invisible in Activity Monitor contract") for the full list
of techniques and the numeric ship gate.
